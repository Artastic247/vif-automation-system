# WI_027_ORGANISATIONAL_KNOWLEDGE_UPDATE

## Purpose
Update controlled organisational knowledge so validated learning improves future app-development and management-system work.

## Scope
Process knowledge, coding patterns, prompt library, tool routing, AI bad-output knowledge, standards applicability, app-specific knowledge, audit knowledge, contingency knowledge and validation knowledge.

## When to use
Use after lessons learned, RCA/CAPA, audit findings, tool/model changes, standards/customer requirement updates, validator calibration or app-specific discovery.

## When not to use
Do not use to store unverified opinions or proprietary/copyrighted standard text.

## Owner/agent
Organisational Knowledge Controller with relevant artefact owner.

## Inputs
Knowledge source, evidence, affected artefact, owner, approval need, communication need, superseded content and adoption check.

## Method steps
1. Classify knowledge item.
2. Confirm evidence/source status.
3. Identify affected artefacts: process, coding pattern, prompt, tool route, AI output, standard, app, audit, contingency, validation.
4. Update controlled artefact.
5. Mark superseded guidance.
6. Communicate update to affected agents/tools.
7. Verify adoption and recurrence impact.
8. Record knowledge update.

## Outputs/records
Knowledge update record, updated artefact, communication evidence, superseded-item note and adoption verification.

## Evidence required
Source evidence, updated artefact, approval record, communication/adoption evidence and review date.

## Linked ADP processes
ADP-02, ADP-10, ADP-16, ADP-20 and ADP-21.

## Linked MOP/COP/SOP processes
Clause 7 organisational knowledge/documented information, Clause 9 audit and Clause 10 improvement.

## Linked gates
Knowledge update gate, lessons learned gate, improvement gate, prompt/tool gate and maturity gate.

## Tools allowed
Knowledge register, prompt library, coding pattern register, tool routing matrix, standards control, validation reports.

## Tools prohibited
Embedding proprietary standard text, unsupported compliance claims, stale knowledge, unapproved customer-specific requirements.

## Risks
Stale knowledge, repeated error, unsupported claim, wrong tool route, AI bad-output pattern not captured.

## Controls
Knowledge updates require evidence, owner, approval where needed and superseded-content control.

## Interfaces
Knowledge Controller, Prompt Auditor, Coding Pattern Owner, Tool Reviewer, Standards Specialist, QA Gatekeeper.

## PASS/HOLD/BLOCKED rules
PASS when knowledge is updated and adopted. HOLD when review/adoption pending. BLOCKED when obsolete knowledge can enable unsafe work or false claim.

## Escalation
Escalate unsupported compliance claim, stale critical guidance, repeat failure due to old knowledge or unapproved customer source.

## MLA / maturity dependency
M3 requires controlled knowledge updates. M4/M5 require trend/effectiveness evidence.

## Update trigger
Lesson learned, RCA/CAPA, audit finding, standards/tool change, customer requirement, validation failure or support incident.
