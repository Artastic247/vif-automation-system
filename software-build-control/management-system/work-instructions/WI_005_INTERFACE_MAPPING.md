# WI_005_INTERFACE_MAPPING

## Purpose
Map handoffs and interfaces so inputs, outputs, control points, risks and evidence are not lost between workstations or processes.

## Scope
Interfaces across app development, management-system processes, audit, validation, release, support and improvement.

## When to use
Use when two roles, tools, processes or systems exchange information, evidence, code, decisions or responsibilities.

## When not to use
Do not use only as a communication list; every interface must include control and evidence.

## Owner/agent
Process Engineer with receiving process owner.

## Inputs
Sender, receiver, input, output, control point, risk, evidence, handoff requirement, escalation path, maturity dependency and gate impact.

## Method steps
1. Define sender and receiver.
2. Define input and expected output.
3. Define control point and owner.
4. Define risk and mitigation.
5. Define evidence required.
6. Define handoff acceptance rule.
7. Define escalation route.
8. Define maturity dependency and gate impact.
9. Record interface in matrix.

## Outputs/records
Interface map, handoff record, risk/control evidence and gate decision.

## Evidence required
Sender/receiver confirmation, input/output record, evidence record, gate decision and escalation if failed.

## Linked ADP processes
All ADP handoffs from ADP-01 through ADP-21.

## Linked MOP/COP/SOP processes
Clause 4 process interaction, Clause 8 operation, Clause 9 audit and Clause 10 improvement.

## Linked gates
Interface, handoff, verification, release and improvement gates.

## Tools allowed
Interface matrix, handoff checklist, GitHub evidence, job cards, verification records.

## Tools prohibited
Informal handoff without evidence for critical scope.

## Risks
Dropped handoff, wrong receiver, incomplete evidence, hidden risk, protected-scope breach.

## Controls
Critical interfaces require evidence before next process starts. Missing interface evidence creates HOLD.

## Interfaces
All ADP, MOP, COP and SOP process owners plus relevant tools/providers.

## PASS/HOLD/BLOCKED rules
PASS when interface evidence is complete. HOLD when non-critical handoff evidence is missing. BLOCKED when protected or release-critical interface is uncontrolled.

## Escalation
Escalate missing handoff, security/data risk, unapproved release path or repeated interface failure.

## MLA / maturity dependency
M1 draft interface. M2 pilot handoff. M3 released interface. M4/M5 measured interface performance and improvement.

## Update trigger
New interface, process change, tool change, handoff failure, audit finding or RCA/CAPA.
