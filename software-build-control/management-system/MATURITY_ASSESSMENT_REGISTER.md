# MATURITY_ASSESSMENT_REGISTER.md

## Purpose
Record maturity assessments for processes, agents, skills, validators, prompts, workflows, tools and future app onboarding routes.

## Scope
Applies to all governed entities within the Software Build Control System.

## Owner/agent
QA Gatekeeper owns assessment evidence. Process owners own maturity claims. System owner approves M4/M5 automation-related maturity.

## Inputs
Audit results, KPI trends, validation reports, RCA/CAPA effectiveness, lessons learned, evidence audits and management-review decisions.

## Outputs/records
Maturity assessment, automation recommendation, escalation, audit requirement and improvement action.

## Maturity level
Uses M0–M5 maturity model.

## Evidence required
Audit evidence, KPI evidence, RCA/CAPA evidence, effectiveness evidence, validation reports and management-review decisions.

## Linked process
MOP-03 Planning/risk/opportunity, MOP-05 Internal audit, MOP-06 Management review, MOP-07 Corrective action and continual improvement.

## Linked agent/skill/procedure/module
QA Gatekeeper, System owner, Process Engineer, maturity-level model and audit programme.

## Interface/control point
Interfaces with automation governance, audit programme, RCA/CAPA, validators and release readiness.

## PASS/HOLD/BLOCKED rules
- PASS: maturity level supported by evidence and audit.
- HOLD: maturity plausible but evidence incomplete.
- BLOCKED: maturity overstated or unsafe automation/release risk exists.

## PDCA
- PLAN: define maturity criteria.
- DO: perform assessment.
- CHECK: review evidence and effectiveness.
- ACT: revise maturity, escalate or improve.

## Audit frequency
Per audit programme and before automation/maturity increase.

## Automation allowance
Automation follows approved maturity model only.

## Escalation
Escalate maturity overstatement, false PASS or weak evidence.

## Update trigger
Audit result, KPI trend, RCA/CAPA, validator change, AI/prompt issue or management review.
